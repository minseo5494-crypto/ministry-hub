## ★ 2026-07-14 2차 세션 갱신 (이 문서의 전제가 여러 개 바뀜)

### 사업 모델 재정의 (대표 확인)

**"권리사와 협상해서 라이선스를 얻는다"가 아니라, "협상이 필요 없는 구조로 바꾼다"가 목표다.**

- 팀이 **자기들이 쓰는 악보를 직접 업로드**하고, **그 팀 멤버만** 열람한다.
- 팀 밖으로 나가지 않으므로 공중송신/불법 유통에 해당하지 않는다는 것이 이 모델의 논리.
- WORSHEEP은 유통자가 아니라 **저장 공간을 빌려주는 호스팅 사업자** 포지션.
- 원래의 "모두에게 악보를 쉽게 제공한다"는 비전은 권리사 협상이 선행돼야 하므로 **이번 전환에서 제외**.
- 요금제는 당분간 **전부 무료** 전제로 진행 (Free 요금제 재설계 항목 폐기).

> **⚠️ 이 논리는 기술적 격리가 실제로 작동할 때만 참이다.** 구멍이 하나라도 남으면
> "팀끼리만 본다"는 주장 자체가 무너진다. 즉 아래 기술 항목들은 부수 작업이 아니라
> **모델의 성립 조건**이다. (특히 Storage 버킷)

### 모델과 현재 구조가 충돌하는 지점 ⚠️

1. **곡의 99%를 운영자가 올렸다.** 1,811곡 = 조민서(admin), 19곡 = zion Park. 전부 `고페르` 팀 공유.
   "팀이 자기 악보를 올린다"가 아니라 "운영자가 라이브러리를 구축했다"로 읽힐 여지가 크다.
2. **`upload/page.tsx`의 관리자 벌크 업로드 도구가 남아 있다.** 운영자 직접 유통의 증거가 되므로
   제거 또는 봉인 필요.
3. **코드 밖 항목**: 이용약관에 "업로드 권리 책임은 업로더에게" 명시, 권리사 신고 시 takedown 절차,
   반복 침해 팀 제재 규정. 이게 있어야 호스팅 사업자 포지션이 성립.

### 1단계 완료 — anon 차단 (2026-07-14 적용됨)

마이그레이션: `supabase/migrations/20260714_team_isolation_step1_anon_lockdown.sql` (롤백 SQL 동봉)

원인은 PERMISSIVE 정책이 OR로 합쳐지는 것이었다. `qual=true` 짜리 하나가 올바른 정책을 통째로 무력화했다.

| 삭제한 정책 | 문제 |
|---|---|
| `songs` / `Enable read access for all users` | `qual=true`, `{public}` → **비로그인 포함 전원에게 전곡 개방** |
| `songs` / `Authenticated users can view songs` | `auth.role()='authenticated' OR visibility='public'` → 로그인만 하면 전곡 |
| `song_sheets` / `Public sheets are viewable by everyone` | `visibility` 기본값이 `'public'`이라 1,672건 전량 개방 |
| `shared_setlists` / `shared_setlists_select` | `{public}` + `status='active'` → 비로그인에게 공유 콘티 노출 |
| `shared_setlist_comments` / `comments_select` | `qual=true`, `{public}` |
| `song_likes` / `Users can view all likes` | `qual=true`, `{public}` → 곡 ID·유저 ID 유출 |

`songs_select`는 "전체공개 완전 폐지" 방침에 맞춰 재작성 (`visibility='public' OR IS NULL` 우회로 제거).
`songs.visibility` 기본값 `'public'` → `'teams'`, `song_sheets.visibility` → `'private'`.

**검증 (anon 실측, 적용 전 → 후)**: songs 1,830 → **0** · song_sheets 1,672 → **0** ·
shared_setlists 2 → **0** · song_likes → **0**. 곡 데이터 직접 요청 시 `[]` 반환.
로그인 상태(조민서/고페르) 기능 4종 정상 동작 확인 완료.

## 0. 핵심 결론 (한 줄)

## 1. 라이브 실측 결과 (anon 키 = 비로그인 방문자 시점)

프로젝트의 공개 anon 키로 로그인 없이 Supabase REST를 직접 호출한 결과:

| 대상 | anon(비로그인) 접근 | 판정 |
|---|---|---|
| `songs` | **1,830곡 전부 읽힘** | 🔴 노출 |
| `song_sheets` | **1,672행 전부 읽힘** | 🔴 노출 |
| `songs.file_url` → 실제 파일 | **HTTP 206 성공 (PNG/PDF 다운로드됨)** | 🔴 노출 |
| `teams` | 0건 (차단) | ✅ 정상 |
| `team_members` | 0건 (차단) | ✅ 정상 |
| `team_setlists` | 0건 (차단) | ✅ 정상 |
| `users` | 0건 (차단) | ✅ 정상 |
| `feedbacks` | 0건 (차단) | ✅ 정상 |
| `shared_setlists` | 2건 읽힘 | 🟠 크로스팀 노출 |

**결정적 사실**: 현재 songs 1,830곡은 **이미 전부 `visibility='teams'`** 로 되어 있다 (public=0, private=0, NULL=0). 즉 "전체공개를 팀공개로 바꾸는" 데이터 작업은 **이미 되어 있다.** 그런데도 비로그인 방문자가 1,830곡을 전부 읽는다 = **`songs`/`song_sheets`의 RLS가 visibility·팀 멤버십을 전혀 강제하지 않는다.** 파일 저장 버킷(`song-sheets`, `sheetmusic`)도 public이라 URL만 알면 파일이 그대로 받아진다.

→ **결론: visibility 컬럼을 바꾸는 게 문제가 아니다. RLS 정책과 Storage 버킷이 열려 있는 게 문제다.** UI만 팀 스코프로 바꾸고 이 둘을 안 고치면 라이선스 구멍은 100% 그대로 남는다.

## 2. 현재 구조 (조사 요약)

### 데이터 계층
- `songs`: `team_id` 없음. `visibility('private'|'teams'|'public')` + `shared_with_teams[]` + `owner_type/owner_id` + `is_hidden` + `upload_status` 보유. 실제 악보 파일은 `songs.file_url`.
- `song_sheets`: `team_id` + `visibility` 컬럼 존재하나 **코드 전체에서 SELECT 0건 = 사실상 미사용**. (혼동 주의: "이미 팀 컬럼 있음"은 이 미사용 테이블 얘기)
- 팀 인프라는 이미 성숙: `teams`(invite_code), `team_members`(role/status), `team_setlists`/`team_setlist_songs`, `TeamRolesManager`, `teams/create`, `teams/join`, `my-team/[id]`, `my-team/[id]/settings`(승인/강퇴/역할).

### 접근 제어
- `src/middleware.ts`: next-intl 로케일 처리만. **인증 가드 전무.** 서버/엣지 레벨 접근제어 없음.
- `src/utils/supabase/` 디렉토리 없음, `@supabase/ssr` 미사용. `src/lib/supabase.ts`의 anon 클라이언트 하나를 브라우저/서버 공용. **실질 신뢰경계 = RLS 뿐** — 그런데 그 RLS가 songs에서 안 먹힘(위 실측).
- `main/page.tsx:807-878 fetchSongs()`: 로그인 가드 없이 `.select('*')` 전체 조회 후 **JS로만** visibility 필터링. RLS가 뚫려 있으므로 네트워크 탭에는 팀 악보가 그대로 내려옴.

### 유출 경로 (security 감사 + 실측)
1. 🔴 **[치명적] Storage 버킷 public** — `song-sheets`, `sheetmusic` 둘 다 public read. `getPublicUrl()` 사용 10곳, `createSignedUrl` 0곳. RLS 바꿔도 파일은 샌다. (실측으로 HTTP 206 확인)
2. 🔴 **[치명적] songs/song_sheets RLS 무력** — anon이 1,830곡 전부 조회. (실측)
3. 🟠 **[높음] pending 멤버 통과** — 오래된 RLS(`20241212_song_sheets.sql:41-47`, `20241214_team_roles_permissions.sql`)가 `status='active'`를 체크 안 함. 초대코드만 알면 승인 전에도 팀 콘텐츠 열람. (최신 마이그레이션엔 조건 있음 → 정책 혼재)
4. 🟠 **[높음] explore/shared_setlists 크로스팀 공유** — `20260305_shared_setlists_community.sql:174-179` SELECT 정책이 `auth.uid() IS NOT NULL`만. 로그인만 하면 남의 팀 콘티(곡·편곡 메타 스냅샷) 열람. 설계 목적 자체가 "팀 간 공유"라 팀 격리와 정면충돌.
5. 🟠 **[중간] Service Worker PDF 7일 캐시** — `next.config.js` PDF CacheFirst. signed URL로 바꿔도 한 번 본 파일은 로컬에 7일 잔존.
6. 🟠 **[중간] `/api/songs/weekly-popular`** — service_role(RLS 우회)인데 인증 검증 없음. 누구나 GET. 크로스팀 집계.
7. 🟢 [낮음] SEO: sitemap/robots/OG 없음, 전 페이지 `'use client'`라 SSG 노출은 적음. AI 검색은 검색어 문자열만 외부 전송 → 문제 아님.

## 3. 전환 설계

### 3-A. DB / RLS (최우선, 필수)
1. **`songs` SELECT RLS 재작성** — anon 차단, 로그인 사용자는 `visibility='teams' AND 소속 active 팀` 또는 본인 private/공개 라이선스분만. (현재 정책이 사실상 전체 허용 상태로 추정 → 라이브 `pg_policies`로 실제 정책 확인 후 교체)
2. **`song_sheets` 동일 적용** + 미사용이면 정리 여부 결정.
3. **팀 멤버십 조건에 `status='active'` 통일** — 오래된 정책들(위 유출 3번) 일괄 수정.
4. **Storage 버킷 private 전환** — `song-sheets`, `sheetmusic`를 Private로. `sheetmusic_select_public` 정책 삭제.
5. **`file_url` 스키마 변경** — 버킷 private 전환 시 기존 `getPublicUrl` 기반 URL이 전부 깨짐. `file_url`(전체 URL)이 아니라 **path만 저장**하고, 조회 시 서버 API가 권한 확인 후 `createSignedUrl` 발급하도록.

### 3-B. 코드 (RLS와 반드시 함께)
- `getPublicUrl()` 10곳(`upload/page.tsx`, `main/page.tsx:1451`, `admin/*`, `my-page/page.tsx`, `useNotebooks.ts`, `AddPageModal.tsx`) → 서명 URL 발급 API 경유로 교체.
- `main/page.tsx fetchSongs()`: 클라이언트 전량 조회 폐지 → 팀 스코프 서버 쿼리.
- `/api/songs/weekly-popular`: 인증 + 팀 스코프 집계, 또는 폐지.
- `explore` + `shared_setlists`/`useCommunity`: **크로스팀 공유 기능 폐지 또는 "내 팀만" 스코프로 전면 재설계** (라이선스 위반 소지 1순위).
- `next.config.js`: PDF 캐시 정책 축소 + 권한 변경 시 캐시 무효화.
- `AddSongModal.tsx`: "전체 공개" 라디오 제거, `visibility` 기본값 `'teams'`, 팀 없는 사용자 업로드 차단.
- `upload/page.tsx`(관리자 벌크 도구): **관리자 가드 존재 여부 미확인 → 반드시 확인**. `visibility` 기본값 `'public'` → 변경.

### 3-C. 메인 페이지 재설계 (추천: 팀 대시보드형)
검색바 중심 홈을 "내 팀 악보함 + 팀 콘티" 중심으로 재편. 기존 `TeamSharedSection`/`fetchTeamSetlists` 재사용으로 배관 절반은 이미 깔림.

- **비로그인**: 검색바 없는 랜딩 + 로그인 CTA만. ("자유 검색"이 존재하지 않음을 명확히)
- **로그인+팀 있음**: 상단 팀 전환기 → "내 팀 악보함에서 검색" 스코프 검색 → 다가오는 콘티 → 우리 팀 악보함.
- **로그인+팀 없음**: `my-team` 빈 상태 패턴 이식 → "팀 만들기 / 초대코드로 참여". 악보함·검색 비활성 + "팀에 가입하면 열립니다".

검색 카피(`messages/ko.json:198-200` "지금 바로 검색하세요")·온보딩(`OnboardingGuide`, `ko.json:70-72` "악보 검색")도 팀 스코프로 수정. AI 검색은 이미 로그인 필수라 스코프만 상속시키면 유지 가능.

### 3-D. 마이그레이션 전략
- songs는 이미 전량 `visibility='teams'`, `is_hidden=false`. 남은 건 **각 곡을 어느 팀에 귀속시키느냐** — 현재 `songs`에 `team_id`가 없고 `shared_with_teams[]`/`owner_id`로 표현. 업로더가 팀이 없거나 다중 팀이면 귀속 기준이 모호 → **고아 데이터 처리 규칙 필요** (아래 4번 결정에 종속).
- 유예: 기존 사용자 콘텐츠 즉시 삭제 금지. 30일 "팀 공개 자동 전환 + 업로더 개별 안내" 후 미조치 시 비공개.

   - "1,809곡" 트랙션 서술 소멸 → 사업계획서 경쟁우위 축을 "협업 워크플로우/저작권 준법성"으로 이동.

2. **기존 1,809곡 처리: 업로더 기준 팀 자동 귀속** — `uploaded_by`가 있는 곡은 해당 유저의 활성 소속 팀(`team_members WHERE status='active'`)으로 자동 귀속하는 스크립트 실행.
   - ⚠️ **단서 (backend 실측)**: `song_sheets` 1,672건은 샘플 전부 `uploaded_by=NULL`, `songs`도 일부는 업로더가 팀 미소속이거나 다중 팀일 수 있음. 이런 **완전 고아/모호 케이스는 자동 귀속 불가** → 비공개 격리(RLS 차단) 후 재검수 큐로 보내고, 확인된 것만 수동 배정. 즉 "업로더 기준 자동 귀속"이 1순위 규칙, 격리가 폴백.
   - 자동 귀속으로도 저작권 리스크 자체가 사라지진 않음(노출 범위만 축소) — 재검수 단계에서 라이선스 확인 병행 권고.

3. **explore + shared_setlists 크로스팀 공유: 내 팀 스코프로 축소** — 기능은 유지하되 로그인 필수 + 내 소속 팀 콘텐츠만 노출하도록 전면 재설계. (폐지가 아니라 축소)
   - 우선 shared_setlists가 비로그인에도 읽히는 RLS 드리프트 버그부터 수정.
   - "인기 악보"는 팀 스코프 집계로, "공유 콘티"는 내 팀 셋리스트 범위로 제한.

4. **긴급 지혈: 전체 계획에 통합 진행** — 별도 선행 작업으로 분리하지 않고 전환 작업의 일부로 함께 처리. (현재 프로젝트가 냉동/중단 상태라 실사용자 노출 리스크가 낮다는 판단. 단, RLS anon 차단 + 버킷 private 전환은 착수 시 가장 먼저 배치.)
