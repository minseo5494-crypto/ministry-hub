# 독립 필기노트 서비스 — 수정 계획서

> 작성일: 2026-02-25
> 목적: 콘티(셋리스트)와 분리된 독립 필기노트 서비스 구현
> 기반: GoodNotes 통합 계획서 분석 → 현실적 축소 버전

---

## 1. 개요

### 1-1. 현재 문제

- 콘티에서 필기를 저장하면 `setlist_notes` 테이블에 저장되고, my-page에서는 **읽기 전용**으로만 표시
- 필기노트를 독립적으로 수정하거나 페이지를 추가하는 기능이 없음
- 콘티와 필기노트가 동일 데이터를 참조하여 구분이 모호

### 1-2. 목표

- 콘티에서 "저장" 시 데이터를 **복사**하여 독립적인 `notebooks` 테이블에 저장
- 필기노트에서 자유롭게 수정, 페이지 추가, 악보 추가 가능
- 콘티 원본에는 영향을 주지 않음 (완전 독립)

### 1-3. 핵심 원칙

- 콘티 저장 → 필기노트에 **복사본** 생성 (이미 있으면 사용자에게 업데이트 여부 확인)
- 필기노트는 콘티와 완전히 **독립** (수정해도 콘티에 영향 없음)
- 기존 SheetMusicEditor는 **최소 수정** (새 prop 2~3개 추가)
- 기존 `sheet_music_notes`, `setlist_notes` 테이블은 그대로 유지

## 2. 데이터 흐름 변경

### 2-1. 현재 흐름

```
콘티 → setlist_notes 저장 → my-page에서 읽기 전용 표시
```

### 2-2. 변경 후 흐름

```
콘티 → setlist_notes 저장 (기존 유지)
         ↓ 복사
       notebooks 테이블에 독립 복사본 생성
         ↓
       my-page에서 편집 가능 (페이지 추가/삭제/악보 추가)
```

- 콘티에서 수정 → 필기노트에 영향 없음 (복사본)
- 필기노트에서 수정 → 콘티에 영향 없음 (독립)

### 2-3. 콘티 재저장 시 처리

콘티를 다시 저장할 때 이미 복사된 노트북이 있으면:

```
"기존 필기노트를 업데이트할까요?"
  [업데이트] — 콘티 최신 데이터로 덮어쓰기 (필기노트에서 수정한 내용 사라짐)
  [새로 추가] — 별도 노트북으로 생성
  [건너뛰기] — notebooks 업데이트 안 함
```

## 3. DB 변경: 새 테이블 1개

### 3-1. `notebooks` 테이블

```sql
CREATE TABLE notebooks (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  user_id UUID NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  title TEXT NOT NULL DEFAULT '제목 없는 노트',

  -- 출처 추적 (콘티에서 복사된 경우)
  source_setlist_id UUID,           -- 원본 콘티 ID (없으면 직접 생성한 노트)
  source_setlist_title TEXT,        -- 원본 콘티 제목 (표시용)
  team_id UUID,                     -- 팀 ID (콘티 연동 시)

  -- 페이지 데이터 (JSONB 배열 — 각 원소가 하나의 페이지)
  pages JSONB NOT NULL DEFAULT '[]',

  -- 메타
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now(),
  deleted_at TIMESTAMPTZ
);

-- 인덱스
CREATE INDEX idx_notebooks_user ON notebooks(user_id) WHERE deleted_at IS NULL;
CREATE INDEX idx_notebooks_user_updated ON notebooks(user_id, updated_at DESC) WHERE deleted_at IS NULL;
CREATE INDEX idx_notebooks_source_setlist ON notebooks(user_id, source_setlist_id) WHERE deleted_at IS NULL;

-- RLS
ALTER TABLE notebooks ENABLE ROW LEVEL SECURITY;
CREATE POLICY "notebooks_select" ON notebooks FOR SELECT USING (auth.uid() = user_id);
CREATE POLICY "notebooks_insert" ON notebooks FOR INSERT WITH CHECK (auth.uid() = user_id);
CREATE POLICY "notebooks_update" ON notebooks FOR UPDATE USING (auth.uid() = user_id);
CREATE POLICY "notebooks_delete" ON notebooks FOR DELETE USING (auth.uid() = user_id);

-- updated_at 자동 갱신
CREATE TRIGGER trigger_notebooks_updated_at
  BEFORE UPDATE ON notebooks
  FOR EACH ROW
  EXECUTE FUNCTION update_sheet_music_notes_updated_at();
```

### 3-2. 왜 별도 notebook_pages 테이블을 안 만드는가

- 이전 분석에서 **RLS 서브쿼리 성능 문제**가 지적됨 (notebook_pages에서 매번 notebooks를 서브쿼리)
- `user_id`가 notebooks에 직접 있으므로 RLS가 단순 비교 (`auth.uid() = user_id`)
- 기존 `setlist_notes.note_data`도 JSONB로 곡 데이터 통합 저장 — 동일 패턴
- 한 노트북의 페이지 수는 보통 5~20개 수준으로 JSONB로 충분

## 4. TypeScript 타입 정의

### 4-1. 새 파일: `src/types/notebook.ts`

```typescript
import { PageAnnotation } from '@/lib/supabase'
import {
  SongFormStyle,
  PartTagStyle,
  PianoScoreElement,
  DrumScoreElement,
} from '@/components/SheetMusicEditor/types'

// 노트북 내 개별 페이지
export interface NotebookPage {
  id: string                    // 페이지 고유 ID (UUID)
  pageType: 'sheet' | 'blank'   // 악보 페이지 or 빈 페이지
  order: number                 // 페이지 순서 (0부터)

  // 악보 페이지인 경우
  songId?: string               // songs 테이블 참조
  songName?: string
  teamName?: string
  fileUrl?: string              // 악보 PDF/이미지 URL
  fileType?: 'pdf' | 'image'
  songForms?: string[]

  // 필기 데이터 (기존 PageAnnotation[] 호환)
  annotations: PageAnnotation[]
  songFormEnabled: boolean
  songFormStyle: SongFormStyle
  partTags: PartTagStyle[]
  pianoScores?: PianoScoreElement[]
  drumScores?: DrumScoreElement[]
}

// 노트북 전체
export interface Notebook {
  id: string
  user_id: string
  title: string
  source_setlist_id?: string
  source_setlist_title?: string
  team_id?: string
  pages: NotebookPage[]
  created_at: string
  updated_at: string
  deleted_at?: string | null
}
```

### 4-2. 변환 함수 (SetlistNoteData → NotebookPage[])

```typescript
function convertSetlistNoteToPages(noteData: SetlistNoteData): NotebookPage[] {
  return Object.entries(noteData)
    .sort(([, a], [, b]) => (a.order ?? 999) - (b.order ?? 999))
    .map(([songId, data], index) => ({
      id: crypto.randomUUID(),
      pageType: 'sheet' as const,
      order: index,
      songId,
      songName: data.song_name,
      teamName: data.team_name,
      fileUrl: data.file_url,
      fileType: data.file_type,
      songForms: data.songForms,
      annotations: data.annotations || [],
      songFormEnabled: data.songFormEnabled ?? false,
      songFormStyle: data.songFormStyle || { x: 50, y: 10, fontSize: 24, color: '#000000', opacity: 1 },
      partTags: data.partTags || [],
      pianoScores: data.pianoScores,
      drumScores: data.drumScores,
    }))
}
```

## 7. 페이지 추가/삭제 흐름

### 7-1. 페이지 추가

```
"+" 버튼 클릭 → 팝오버: "빈 페이지" / "악보 추가"
  ├─ "빈 페이지":
  │   → onAddPage(currentIndex, { type: 'blank' })
  │   → useNotebooks.addPage() → pages JSONB에 빈 NotebookPage 추가
  │   → songs 배열에 빈 EditorSong 추가 → 에디터 자동 전환
  │
  └─ "악보 추가":
      → 곡 검색 모달 열기 → 곡 선택
      → onAddPage(currentIndex, { type: 'sheet', songId, fileUrl, ... })
      → useNotebooks.addPage() → pages JSONB에 악보 NotebookPage 추가
      → songs 배열에 EditorSong 추가 → 에디터 자동 전환
```

### 7-2. 페이지 삭제

```
현재 페이지에서 "삭제" 버튼 클릭
→ confirm("이 페이지를 삭제할까요?")
→ onRemovePage(currentSongIndex)
→ useNotebooks.removePage() → pages JSONB에서 해당 인덱스 제거
→ songs 배열 업데이트 → 에디터가 이전/다음 페이지로 전환
```

### 7-3. 저장

```
저장 버튼 클릭 → onSaveAll (기존 다중 곡 저장 로직 그대로)
→ 부모 컴포넌트(my-page)에서 dataList 수신
→ useNotebooks.updateNotebook(notebookId, pages) 호출
→ notebooks 테이블 UPDATE (pages JSONB 갱신)
```

## 9. 위험 요소

| 위험 | 심각도 | 대응 |
|------|--------|------|
| **JSONB pages 크기 초과** | 중 | 페이지 50개 제한. 필기량이 많으면 수 MB까지 커질 수 있으나 PostgreSQL JSONB는 1GB까지 가능하므로 당분간 문제없음. 장기적으로 annotations를 별도 storage 분리 고려. |
| **콘티 재저장 시 덮어쓰기** | 중 | 반드시 confirm 다이얼로그로 확인. "업데이트" / "새로 추가" / "건너뛰기" 3가지 선택지 제공. |
| **SheetMusicEditor 코드 추가** | 중 | `notebookMode` 조건부 렌더링(`{notebookMode && ...}`)으로 최소 80줄만 추가. 기존 로직 변경 없음. |
| **빈 페이지 캔버스 렌더링** | 낮 | fileUrl 없을 때 빈 캔버스 표시하도록 조건 분기 필요. 기존 `loadImage`/`loadPDF` 에러 핸들링 확인. |
| **song_id 타입 (UUID vs TEXT)** | 낮 | notebooks.pages의 songId를 TEXT로 저장하면 기존 패턴과 동일. |
| **자동저장 미구현** | 낮 | 이번 범위에서는 수동 저장 유지. 향후 15~30초 디바운스 자동저장 추가 가능. |
