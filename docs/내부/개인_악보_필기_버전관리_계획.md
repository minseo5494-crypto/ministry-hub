## 1. 배경 및 문제 정의

### 현재 사용자 Pain Point

예배팀에서 인도자가 콘티(셋리스트)에 곡을 넣으면, 각 팀원(피아노, 기타, 드럼, 보컬 등)은 대표 악보 위에 본인만의 필기를 추가해야 한다. 다음에 같은 곡이 콘티에 다시 올라왔을 때, 이전에 필기했던 악보를 손쉽게 가져오고 싶다. 또한 같은 곡이 다른 키/아티스트로 여러 개 등록되어 있을 때 콘티에서 쉽게 교체하고 싶다.

### 현재 시스템의 문제점

| 문제 | 설명 |
|------|------|
| **중복 업로드** | 같은 곡의 악보가 악기별/싱어별로 따로 업로드되어 곡 목록 폭증 |
| **수동 매칭** | 콘티에 곡이 들어가면, 팀원은 본인 수정본을 직접 검색 → PDF 다운로드 → 별도 합침 필요 |
| **필기 재사용 불가** | 다음에 같은 곡이 콘티에 나와도 이전 필기를 쉽게 가져올 수 없음 |
| **곡 버전 미연결** | 같은 곡의 다른 키/아티스트 버전이 완전 별개 레코드로 관리됨 |
| **콘티 내 교체 불편** | 곡의 키/버전을 바꾸려면 삭제 후 재추가해야 함 (key_transposed UI 없음) |

## 2. 현재 시스템 분석

### 관련 DB 테이블 현황

| 테이블 | 현황 | 비고 |
|--------|------|------|
| `songs` | 1곡 = 1악보(`file_url`) 구조 | 곡 간 연결(group_id 등) 없음 |
| `song_sheets` | **타입만 정의, CRUD 미구현** | label, is_primary, visibility, team_id 필드 존재 |
| `sheet_music_notes` | 개인 필기 저장 중 | song_id(TEXT) + user_id별 독립, soft delete |
| `personal_setlist_views` | 콘티 개인화 존재 | customizations(JSONB), replace/insert 지원 |
| `team_setlist_songs` | 콘티 내 곡 순서 | **key_transposed 필드 존재하나 변경 UI 없음** |
| `team_members` | parts[] 필드 존재 | 자유 텍스트 배열, 활용도 낮음 |

| 인프라 | 상태 | 활용 방안 |
|--------|------|-----------|

- 콘티에서 key_transposed 변경 UI
- "내 필기 불러오기" UI 버튼

## 3. 아키텍처 방향 결정

### 결정 1: 곡 버전 관리 → `song_groups` 도입

**현재**: "주의 이름 높이며(C) - Welove"와 "주의 이름 높이며(E) - 마커스"가 완전 별개 레코드
**제안**: `song_groups` 테이블로 느슨한 그루핑

```
song_groups (새 테이블)
├── id: UUID (PK)
├── canonical_name: TEXT     -- "주의 이름 높이며" (정규화된 대표 이름)
├── created_at: TIMESTAMPTZ
└── updated_at: TIMESTAMPTZ

songs (변경)
└── + group_id: UUID? REFERENCES song_groups(id) ON DELETE SET NULL
```

- `group_id`가 nullable → 기존 데이터 100% 호환
- 곡의 독립성 유지 (각 song은 고유한 key, team_name, file_url)
- "같은 곡의 다른 버전" 관계만 표현

### 결정 2: 콘티 내 교체 → 인라인 교체 UI

**현재**: 곡 교체 = 삭제 → 재추가 (key_transposed, notes, selected_form 소실)
**제안**: 인라인에서 같은 그룹의 다른 버전으로 song_id 교체 (메타데이터 유지)

### 결정 3: 필기 불러오기 → 수동 (자동 아님)

**방식**: 콘티에서 "내 필기 불러오기" 버튼 → 사용자가 원하는 필기를 직접 선택
기존 `usePersonalSetlistView`의 `replaceSongWithNote()` 로직 그대로 활용

### Phase 2: 곡 그루핑 + 버전 교체 UI

**목표**: 같은 곡의 다른 키/아티스트 버전을 연결하고 콘티에서 쉽게 교체

#### 콘티 곡 교체 UX

```
곡 행에서 [교체] 버튼 (또는 ... 메뉴) 클릭
→ 모달:
  ┌─────────────────────────────────────┐
  │ "주의 이름 높이며" 다른 버전          │
  │                                     │
  │ 📄 같은 곡 다른 버전 (group)          │
  │  ├ ✅ 현재: Welove (C)              │
  │  ├ 마커스워십 (E) [선택]             │
  │  └ Hillsong (G) [선택]              │
  │                                     │
  │ 🔍 전체 검색 (다른 곡으로 교체)       │
  │  [곡 검색 입력...]                   │
  └─────────────────────────────────────┘

→ "같은 곡 다른 버전" 선택 시:
  - team_setlist_songs.song_id만 교체
  - key_transposed, notes, selected_form 유지

→ "전체 검색" 사용 시:
  - song_id 교체 + key_transposed 초기화
```

#### DB 변경

```sql
-- song_groups 테이블 생성
CREATE TABLE IF NOT EXISTS song_groups (
  id UUID DEFAULT gen_random_uuid() PRIMARY KEY,
  canonical_name TEXT NOT NULL,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);
CREATE INDEX idx_song_groups_canonical_name ON song_groups(canonical_name);

-- songs에 group_id 추가
ALTER TABLE songs ADD COLUMN IF NOT EXISTS group_id UUID
  REFERENCES song_groups(id) ON DELETE SET NULL;
CREATE INDEX idx_songs_group_id ON songs(group_id);

-- RLS
ALTER TABLE song_groups ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Authenticated users can view song groups"
  ON song_groups FOR SELECT
  USING (auth.uid() IS NOT NULL);

CREATE POLICY "Authenticated users can create song groups"
  ON song_groups FOR INSERT
  WITH CHECK (auth.uid() IS NOT NULL);

CREATE POLICY "Admins can update song groups"
  ON song_groups FOR UPDATE
  USING (EXISTS (SELECT 1 FROM users WHERE id = auth.uid() AND is_admin = true));

CREATE POLICY "Admins can delete song groups"
  ON song_groups FOR DELETE
  USING (EXISTS (SELECT 1 FROM users WHERE id = auth.uid() AND is_admin = true));
```

#### 기존 데이터 마이그레이션 (관리자 수동 검토 후 실행)

```sql
-- 같은 song_name의 곡들을 자동 그루핑 (실행 전 SELECT로 검토 필수!)
WITH grouped AS (
  SELECT DISTINCT ON (lower(trim(song_name)))
    lower(trim(song_name)) as normalized_name,
    song_name as canonical
  FROM songs
  WHERE song_name IS NOT NULL
  GROUP BY lower(trim(song_name)), song_name
  HAVING COUNT(*) > 1
)
INSERT INTO song_groups (canonical_name)
SELECT canonical FROM grouped;

UPDATE songs s
SET group_id = sg.id
FROM song_groups sg
WHERE lower(trim(s.song_name)) = lower(trim(sg.canonical_name));
```

## 7. 주의사항

### 기술적 주의
- `sheet_music_notes.song_id`는 TEXT 타입 → songs.id(UUID)와 조인 시 `::text` 캐스팅 필요
- localStorage + Supabase 이중 저장 구조 유지
- 자동 그루핑 마이그레이션은 관리자 수동 검토 필수 ("Amazing Grace" vs "Amazing Grace(새번역)" 같은 미묘한 케이스)
- 그룹 관리를 위한 관리자 어드민 기능 필요 (잘못 연결된 그룹 수정)

### UX 주의
- iOS Safari: 키 변경은 네이티브 `<select>` 사용 권장 (font-size 16px 필수)
- 필기 불러오기는 **수동** (자동 아님) - 사용자가 원할 때만
- 콘티에서 곡 교체 시 key_transposed, notes 등 메타데이터 유지 여부 명확히 표시

### RLS 정책
- `song_groups`: 인증 사용자 읽기/생성, 관리자만 수정/삭제
- 기존 테이블 RLS: 변경 없음
