# 기능 계획 — 악보 → 코드악보 자동 변환

**작성**: 2026-07-21 (5차 세션)
**상태**: 설계 (구현 착수 전 범위·데이터 확정용)

---

## 1. 목표

사용자가 **전체 악보**(오선·음표 포함, PDF/이미지)를 업로드하면, **오선·음표를 걷어내고
마디 단위로 "코드 + 가사"만 남긴 간소화 코드악보**로 자동 변환한다. 인도자·싱어 등
음표가 필요 없는 사람은 이 간소화 뷰만 본다.

### 목표 레이아웃 (참고: `악보comethru_1.jpg`, "싱잉기타" 스타일)
- 한 줄 = 4마디(기본), 마디 경계는 세로선/구획선
- **마디 위**: 코드 (예: `CM7` `Am7` `F` `G`) — 한 마디에 여러 코드 가능(`Am7 G FM7`)
- **마디 아래**: 그 마디에 해당하는 가사 조각
- 좌측에 마디 번호(1, 5, 9, 13…), 박자표(4/4), 반복기호(｜: :｜)·엔딩(1.) 표기

---

## 2. 현실적 접근 — "AI 추출 + 사람 교정" (완전 자동 아님)

> ⚠️ 임의의 악보 이미지에서 100% 자동 변환은 불가능(범용 OMR 미해결 영역).
> 정확도를 정직하게 전제하고 **교정 UI를 필수 포함**한다.

| 입력 악보 유형 | 기대 정확도 | 비고 |
|---|---|---|
| 리드시트 / 코드+가사 악보 (워십 악보 대부분) | 70~90% | 코드·가사가 텍스트로 인쇄 → 비전 LLM 추출 양호 |
| 피아노/성악 풀 스코어 (음표 빽빽) | 낮음 | 코드 심볼 없으면 화성 추론까지 필요 → MVP 범위 밖 |

### 파이프라인
```
업로드 악보(이미지/PDF)
  → [비전 LLM] 구조 추출 → ChordChart JSON
  → [교정 UI] 사용자가 코드/가사/마디 훑어 수정
  → [렌더러] 마디 그리드(HTML/SVG)로 표시 + PDF 저장
```

- **비전 모델**: `@anthropic-ai/sdk` 재사용(이미 설치). 검색용 `claude-3-haiku`와 별개로
  비전 품질 위해 **`claude-sonnet-5`**(기본) 사용. 서버 라우트에서 호출(키 노출 방지).
- PDF 입력은 페이지별 이미지로 변환 후 페이지 단위 추출(기존 pdfjs 렌더 재사용 가능).

## 3. 데이터 모델

현재 `SongStructure`(`{섹션:가사}`)로는 부족(마디·코드 없음). 새 구조 정의.

```ts
interface ChordChart {
  time_signature: string        // "4/4"
  key?: string                  // "C" (선택)
  bars_per_line?: number        // 기본 4
  measures: ChordMeasure[]
}
interface ChordMeasure {
  index: number                 // 1부터
  chords: string[]              // ["CM7"] 또는 ["Am7","G","FM7"]
  lyric: string                 // 그 마디 가사 조각 (없으면 "")
  section?: string              // "Verse1" | "Chorus" | "Bridge" ...
  repeat_start?: boolean        // ｜:
  repeat_end?: boolean          // :｜
  ending?: number               // 1, 2 (엔딩 번호)
}
```

### 저장 위치 — 신규 테이블 `song_chord_charts` (권장)
| 컬럼 | 설명 |
|---|---|
| id | uuid |
| song_id | songs.id FK |
| data | jsonb (ChordChart) |
| status | 'draft' \| 'confirmed' (교정 완료 여부) |
| generated_by | 'ai' \| 'manual' |
| created_by / created_at / updated_at | 감사 |

- 이유: 곡당 1개(추후 버전 관리 여지), `songs` 비대화 방지, RLS는 팀 격리(`get_user_team_ids`) 재사용.
- 대안: `songs.chord_chart jsonb` 단일 컬럼 — 더 단순하나 버전·상태 관리 약함.

## 4. UI 흐름

1. 곡 상세/악보 뷰어에 **"코드악보" 탭/버튼** 신설
2. 없으면 **"코드악보 생성"** → 로딩 → 추출 결과를 교정 화면에 표시
3. **교정 화면**: 마디 그리드, 각 마디의 코드 칩(추가/삭제/수정)·가사 인라인 편집·섹션 지정·반복/엔딩 토글
4. **저장**(status: confirmed) → 이후엔 저장된 코드악보를 바로 렌더
5. **렌더 뷰**: 참고 이미지 레이아웃(마디선+코드+가사). **PDF 저장**(기존 `pdfGenerator` 확장 or 신규)
6. **권한**: 팀 멤버만 조회, 생성/교정은 리더/권한자(기존 `useTeamPermissions` 재사용)

---

## 5. 범위 (MVP → 확장)

**MVP**
- [ ] 리드시트/코드악보 대상 비전 추출(서버 라우트) → ChordChart JSON
- [ ] 교정 UI (코드/가사/마디/섹션)
- [ ] 마디 그리드 렌더 (화면)
- [ ] `song_chord_charts` 테이블 + RLS + 훅

**확장(후속)**
- [ ] PDF 저장(참고 이미지 스타일)
- [ ] 반복/엔딩 정밀 렌더
- [ ] 풀 스코어 화성 추론(정확도 낮음, 신중)
- [ ] 조옮김(키 변경 시 코드 자동 이동)

## 6. 미결 · 확인 필요

1. **입력 소스**: 기존 업로드 악보(`songs.file_url`) 대상? 아니면 변환 전용 새 업로드?
2. **비전 모델/비용**: 곡당 1회 변환. sonnet-5 기준 호출 비용 감내 범위? rate limit 필요.
3. **교정 UX 깊이**: 완전 편집기 vs "명백한 오류만 수정" 경량 편집.
4. **저장 구조**: 신규 테이블 vs `songs` 컬럼.
5. **PDF 저장을 MVP에 포함?** (화면 렌더만 먼저 vs 처음부터 PDF)
