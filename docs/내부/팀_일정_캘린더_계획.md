# 팀 일정 캘린더 기능 계획

> 작성일: 2026-07-15 · 상태: **계획 확정 (코드/DB 미변경)**
> 목표: 팀이 매주 반복되는 연습·섬김·예배 일정을 한 곳에 기록하고, 참석/불참·장소를 관리한다.
>
> **확정 결정 (2026-07-15, 대표)**
> - **범위: 앱 내 기능(1~6단계) 전체** 진행 — 반복·참석·콘티 연결 모두 포함.
> - **구글 캘린더 연동: 이번엔 제외** (7단계 → v2로 보류). 5번 섹션은 참고용으로 남김.
> - 반복: **B안(인스턴스 생성), 기본 12주치.** 참석: **셀프 입력 + 리더 대리 지정.** 콘티 연결: **포함.**

## 0. 한 줄 요약

**"팀 콘티(team_setlists)가 '무슨 곡을 하느냐'를 담는다면, 이 기능은 '언제·어디서·누가 모이느냐'를 담는다."**
콘티와 별개의 `team_events` 테이블을 새로 만들고, 예배 일정은 기존 콘티와 연결(선택)한다.

---

## 1. 요구사항 (대표 요청 정리)

1. 매주 토요일/주일마다 반복되는 일정 기록 — **연습시간 / 섬김시간 / 예배 일정** 등 유형 구분
2. 각 일정에 **참석/불참 팀원** 기록
3. **장소** 정보
4. **구글 캘린더 연동**

---

## 2. 조사 결과 (기존 코드/DB 실측)

- **유령 스키마 존재**: DB에 `services`, `service_team_assignments`, `service_checklists`, `service_cue_items`, `service_cue_songs`, `service_templates` 테이블이 RLS까지 걸린 채 있으나 **프론트 코드에서 사용 0건**. 마이그레이션 파일도 없음(대시보드 수기 생성). → 재활용하지 않고 새로 설계. (별도 정리 대상)
- **`team_setlists`** = 콘티. 이미 `service_date`(date), `service_type`, `title`, `theme`, `notes`, `created_by` 보유. **예배 일정 ↔ 콘티 연결 고리로 활용 가능.**
- **`team_members`**: `role`(leader/admin/member) + `status`(active/pending) + `user_id`. 참석 관리 대상.
- **RLS 헬퍼**: `get_user_team_ids()`(active 팀), `get_user_admin_team_ids()`(leader/admin 팀) 재사용 가능 — 새 팀 격리 모델과 정합.
- **Google 로그인**: `signInWithGoogle`이 `access_type:'offline'`, `prompt:'consent'`까지 설정(리프레시 토큰 받을 준비는 됨). 단 **캘린더 scope 미요청**, 이메일/비번 가입자도 존재 → 연동은 **선택(opt-in)** 이어야 함.
- **my-team/[id] 페이지**: 현재 탭이 `setlists | songs` 2개. 여기에 **`calendar` 탭 추가**가 자연스러움.

## 3. 데이터 모델 (신규)

### 3-1. `team_events` — 일정 본체

| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | uuid PK | |
| team_id | uuid FK→teams | 소속 팀 |
| title | text | 일정 제목 (예: "주일 1부 예배", "정기 연습") |
| event_type | text | `practice`(연습) / `serving`(섬김) / `worship`(예배) / `etc` |
| start_at | timestamptz | 시작 시각 |
| end_at | timestamptz null | 종료 시각 (선택) |
| all_day | boolean | 종일 여부 (기본 false) |
| location | text null | 장소 (자유 입력. v1은 문자열) |
| description | text null | 메모 |
| setlist_id | uuid FK→team_setlists null | 예배 일정 ↔ 콘티 연결 (선택) |
| recurrence_group_id | uuid null | 반복 시리즈 묶음 ID (아래 4번) |
| google_sync | boolean | 이 일정 구글 반영 여부 (기본 true) |
| created_by | uuid FK→users | |
| created_at / updated_at | timestamptz | |

### 3-2. `team_event_attendance` — 참석 현황

| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | uuid PK | |
| event_id | uuid FK→team_events (on delete cascade) | |
| user_id | uuid FK→users | |
| status | text | `attending`(참석) / `absent`(불참) / `maybe`(미정) / `no_response`(무응답, 기본) |
| note | text null | 사유/비고 (예: "5분 늦음", "출장") |
| updated_at | timestamptz | |
| — | UNIQUE(event_id, user_id) | 팀원당 일정별 1행 |

> **설계 포인트**: 참석 행은 미리 만들지 않고, 팀원이 상태를 고르는 순간 upsert. 요약은 `team_members`(active) 대비 집계로 계산. "무응답 3명" 같은 표시 가능.

### 3-3. `user_google_calendar` — 구글 연동 토큰 (연동 채택 시)

| 컬럼 | 타입 | 설명 |
|---|---|---|
| user_id | uuid PK FK→users | |
| refresh_token | text | 구글 리프레시 토큰 (서버에서만 사용) |
| calendar_id | text | 대상 캘린더 (기본 'primary') |
| connected_at | timestamptz | |

> ⚠️ 토큰은 민감정보. **service_role로만 접근**하는 서버 API 뒤에 두고, RLS는 anon/authenticated 전면 차단. 클라이언트는 절대 직접 읽지 않음.

## 4. 반복 일정 처리 (핵심 난제)

세 방식 중 **B안(구체 인스턴스 생성) 권장.**

| 안 | 방식 | 장 | 단 |
|---|---|---|---|
| A. RRULE | iCal 표준 문자열 저장 후 조회 시 전개 | 구글과 호환, 무한 반복 | 단일 회차 수정/참석 관리가 복잡 |
| **B. 인스턴스 생성 (권장)** | "매주 토요일 연습" → 향후 N주치 실제 행 생성, `recurrence_group_id`로 묶음 | 회차별 장소·참석·콘티 연결·개별 수정 쉬움 | 미래 무한 아님(주기적 생성 필요) |
| C. 하이브리드 | 규칙 저장 + 필요 시 인스턴스화 | 유연 | 구현 복잡도 최고 |

**B안 상세**
- 생성 시 유형(한 번 / 매주 토 / 매주 주일 / 매주 토·주일) + 기간(예: 12주) 선택 → 그만큼 `team_events` 행 생성, 같은 `recurrence_group_id` 부여.
- 수정/삭제 시 선택지: **이 회차만 / 이후 전체 / 전체**. (구글 캘린더와 동일 UX)
- 12주 이후 자동 연장은 v2 과제(예약 작업 또는 조회 시 부족분 보충).

## 5. 구글 캘린더 연동 — 방식 선택 필요 ⭐

세 갈래이며 복잡도·효과가 크게 다르다. **대표 결정 필요.**

### 방식 ① iCal 구독 링크 (.ics) — 가장 간단, 읽기 전용
- 팀별(또는 사용자별) 비밀 URL 발급 → 구글 캘린더 "URL로 추가"에 붙여넣기.
- **OAuth·토큰·scope 불필요.** 서버가 `.ics` 텍스트만 생성.
- 애플/네이버 캘린더에서도 그대로 구독됨.
- 단점: **단방향(앱→구글), 읽기 전용**, 구글이 몇 시간에 한 번만 갱신(즉시 아님).

### 방식 ② OAuth 푸시 — 즉시 반영, 쓰기
- 사용자가 "구글 캘린더 연결" → 캘린더 scope 동의 → 팀 일정을 사용자 구글 캘린더에 즉시 생성/수정/삭제.
- `google_event_id` 매핑 저장해 변경 전파. 리프레시 토큰 관리 필요.
- 단점: **구글 계정 로그인 사용자만**, 토큰 관리·에러 처리 복잡, Google Cloud 프로젝트에 Calendar API 활성화 + 동의 화면 심사 가능성.

### 방식 ③ 단계적 (권장): ①로 시작 → 나중에 ②
- v1은 iCal 구독으로 빠르게 가치 제공(모든 캘린더 앱 호환, 리스크 낮음).
- 수요 확인 후 v2에서 OAuth 즉시 동기화 추가.

> **연동 없이도 기능은 완결된다.** 캘린더 연동은 부가 레이어이므로, 1~4번(앱 내 일정)을 먼저 완성하고 연동은 마지막에 붙이는 순서를 권장.

## 6. UI/UX 설계

### 위치
- `my-team/[id]` 페이지에 **`일정` 탭 신설** (`setlists | songs | calendar`).
- 메인 대시보드(로그인+팀)에도 "다가오는 일정" 요약 카드 노출 가능(선택).

### 뷰
- **월 그리드 뷰**(데스크톱 기본) + **아젠다/리스트 뷰**(모바일 기본, 다가오는 일정 순).
- 유형별 색상 뱃지: 연습/섬김/예배/기타.

### 일정 카드/상세
- 제목 · 유형 뱃지 · 시간 · 장소 · 참석 요약(○5 ✕2 ?1 ·무응답3) · 연결된 콘티 바로가기.
- **참석 토글**: 각 팀원이 자기 상태(참석/불참/미정) + 사유 입력. 리더는 전체 요약 조회 + 대리 지정 가능.

### 생성/수정 모달
- 유형 · 제목 · 날짜/시간 · 종일 · 장소 · 메모 · 반복(한 번/매주 토/주일/토·주일 + 기간) · 콘티 연결(예배 유형일 때) · 구글 반영 여부.

### 카피/i18n
- `messages/ko.json`·`en.json`에 `calendar` 네임스페이스 신설.
- iOS Safari: 시간 입력 등 폰트 16px+, 터치 44px 준수(프로젝트 규칙).

## 7. 권한 (RLS) — 팀 폐쇄 공유 모델 준수

- `team_events`: SELECT/INSERT/UPDATE/DELETE 모두 **소속 active 팀 멤버만**. 삭제·수정은 작성자 또는 리더(`get_user_admin_team_ids()`)로 제한 검토.
- `team_event_attendance`: 같은 팀 멤버는 SELECT 가능(참석 현황 공유), 본인 행만 INSERT/UPDATE, 리더는 전체 UPDATE.
- `user_google_calendar`: anon/authenticated 전면 차단, 서버(service_role) 전용.
- 모든 신규 테이블 `alter table ... enable row level security` + 마이그레이션 파일로 관리(대시보드 수기 금지).

## 8. 구현 단계 (제안)

| 단계 | 내용 | 산출물 |
|---|---|---|
| **1. DB 스키마** | `team_events` + `team_event_attendance` 마이그레이션 + RLS | `supabase/migrations/2026xxxx_team_events.sql` |
| **2. 타입/훅** | `types/teamEvent.ts`, `useTeamEvents` (CRUD), `useEventAttendance` | 훅·타입 |
| **3. 일정 탭 UI** | my-team/[id] `calendar` 탭 + 월/아젠다 뷰 + 상세 | 컴포넌트 |
| **4. 생성/수정 모달 + 반복(B안)** | 인스턴스 생성, 회차/이후/전체 수정 | 모달 |
| **5. 참석 기능** | 토글 + 요약 집계 + 사유 | UI+훅 |
| **6. 콘티 연결** | 예배 일정 ↔ team_setlists 링크 | 링크 UI |
| **7. 구글 연동** | (선택) iCal 구독 → 이후 OAuth 푸시 | 서버 API |
| **8. 문서/카피** | i18n, HANDOFF 갱신 | — |

> 1~6이 "앱 내 완결" 지점. 7은 별도 붙임. 4·5가 가장 손이 많이 감.
